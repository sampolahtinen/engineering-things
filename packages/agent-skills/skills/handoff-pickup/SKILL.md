# Handoff Pickup

## Quick Start

When given a handoff path such as `/tmp/handoff-topic.md`:

1. Read the handoff artifact first.
2. Identify the requested continuation focus, explicit blockers, and suggested skills.
3. Verify the current repo state before making changes.
4. Continue only the next actionable slice unless the user asks for broader work.

## Workflow

### 1. Load And Triage The Artifact

Extract these fields from the handoff:

- source session purpose and current branch/worktree
- what context is reference-only and should not be redone
- prioritized next actions
- blockers requiring human approval
- gotchas, commands, or repo-specific warnings
- referenced PRs, issues, docs, memory files, or local paths
- suggested skills to load before acting

If the handoff references another artifact by path or URL, read it only when it is needed for the next action.

### 2. Validate Current State

Never assume the handoff is still current. Before editing or reporting status:

- inspect the current working directory and branch
- check `git status` for unrelated user changes
- check referenced PRs, branches, or files when they affect the next action
- confirm referenced blockers are still blockers before treating them as facts
- verify files still exist before deleting, moving, or editing them

If the handoff names a different worktree than the current one, ask the user whether to switch context or continue in the current workspace unless the requested next action is clearly repo-independent.

### 3. Preserve The Handoff Boundary

Respect scope boundaries from the artifact:

- Do not redo completed work marked as reference-only.
- Do not bypass human-review, approval, production, or destructive-operation blockers.
- Do not mix separate follow-up topics into one change unless the handoff explicitly says they belong together.
- Do not delete scripts or migration-like tooling merely because grep shows no imports; confirm retirement when the artifact calls this out.

### 4. Continue The Work

For implementation follow-ups:

- choose the smallest viable next topic from the handoff priority order
- follow repo branch, test, formatting, and PR conventions mentioned in the handoff and project instructions
- re-run the relevant searches instead of trusting stale dead-code or caller analysis
- verify with the lowest-cost meaningful checks before claiming completion

For analysis follow-ups:

- summarize what changed since the handoff was written
- separate confirmed facts from handoff assumptions
- present blockers and next decisions clearly

## Response Pattern

Start by summarizing the handoff in 2-4 bullets:

- continuation focus
- immediate next action
- blockers or approvals needed
- verification you will perform or already performed

Then proceed with the work. If blocked, ask one concrete question with the decision needed.

## Guardrails

- Treat `/tmp` handoff files as local transient context, not durable project documentation.
- Redact secrets if the handoff accidentally includes them.
- Prefer loading suggested skills when their trigger matches the actual next action.
- If the handoff conflicts with current repository instructions, follow current repository instructions and call out the conflict.
